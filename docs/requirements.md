# RFC単語帳アプリ 要件定義書

## 1. プロジェクト概要

### 1.1 アプリ名
**rfc-tangocho**（RFC単語帳）

### 1.2 目的
RFCの番号とその名称を効率よく暗記するためのWebアプリ。
クイズ形式で繰り返し学習することで、RFC番号と仕様名の対応関係を定着させる。

### 1.3 ターゲットユーザー
- ネットワークエンジニア・プロトコル学習者
- RFC仕様を参照する機会が多い開発者
- 技術試験・資格の学習者

### 1.4 デプロイ先
**GitHub Pages**（静的サイト）

## 2. 機能要件

### 2.1 クイズモード（必須・メイン機能）

#### 出題形式
- **RFC番号 → 名称を答える**: RFC番号を提示し、その仕様名を4択から選ぶ
- **名称 → RFC番号を答える**: 仕様名を提示し、RFC番号を4択から選ぶ

#### クイズフロー
1. 出題形式を選択（番号→名称 / 名称→番号）
2. 出題数を選択（10問 / 20問 / 全問）
3. カテゴリを選択（全カテゴリ / 特定カテゴリ）
4. オプション設定:
   - 「廃止済みRFCを含める」チェックボックス（デフォルト: オフ）
   - 「苦手問題のみ出題」チェックボックス（デフォルト: オフ、localStorageに進捗がある場合のみ有効）
5. クイズ開始 → 1問ずつ出題
6. 回答後に正解・不正解をフィードバック（詳細は後述）
7. 全問終了後にスコアを表示（正解数 / 出題数、正解率）
8. 結果画面から再チャレンジ or トップに戻る

#### 回答フィードバック
- 正解時: 選択した選択肢を緑色でハイライト表示
- 不正解時: 選択した選択肢を赤色でハイライト、正解の選択肢を緑色でハイライト
- 正解・不正解に関わらず、正解RFCの `note` が空でない場合は補足説明を表示
- 「次の問題へ」ボタンを表示（キーボードでは Enter / Space で進む）

#### 廃止済みRFCの出題方針
- `obsoletedBy` が非null（廃止済み）のRFCは、デフォルトではクイズの正解候補（出題対象）から除外する
- クイズ設定の「廃止済みRFCを含める」をオンにした場合のみ出題対象に含める
- 廃止済みRFCを誤答選択肢として表示する場合は制限なし

#### 選択肢の生成
- 正解1つ＋同カテゴリから3つをランダムで選ぶ
- 選択肢はシャッフルして表示
- 同カテゴリのRFCが3件未満の場合、不足分は他カテゴリからランダムで補完する（常に4択を維持する）
- 廃止済みRFCも誤答選択肢の候補として使用可能とする

### 2.2 フラッシュカードモード（任意・補助機能）

#### フラッシュカード設定
- カテゴリを選択（全カテゴリ / 特定カテゴリ）
- 廃止済みRFCを含めるか選択（デフォルト: オフ）

#### フラッシュカードの動作
- カードの表面にRFC番号を表示（カテゴリ名を小さく添える）
- クリック/タップで裏返し、名称・shortName・`note` を表示
- 「覚えた」: そのカードをデッキから除外し次のカードへ進む（localStorageの `correct` をインクリメント）
- 「もう一度」: そのカードをデッキの末尾に戻し次のカードへ進む
- デッキ内の全カードが「覚えた」になったら完了画面を表示
- 完了画面から「最初からやり直す」（全カードを再デッキ化）または「トップに戻る」を選択可能

### 2.3 一覧・検索モード（任意）

#### 一覧の表示項目（1件につき）
- RFC番号
- 正式名称（`name`）
- 略称（`shortName`、ある場合のみ）
- カテゴリ（`label` を表示）
- 廃止状況（`obsoletedBy` が非nullの場合「廃止済」バッジを表示）
- 学習進捗（localStorageから取得した正解回数 / 挑戦回数。未挑戦の場合は表示なし）

#### 操作
- キーワード検索（RFC番号 or 名称の部分一致、リアルタイムフィルタ）
- カテゴリフィルタ（複数選択可）
- 廃止済みRFCの表示/非表示切り替え
- RFC番号順 / カテゴリ順 の並び替え

## 3. データ要件

### 3.1 データ管理方針

- **JSONファイルをリポジトリに含めて管理**
- データの追加・修正はPull Requestで行う
- アプリ実行時にJSONをfetchして利用（ビルド不要）

### 3.2 データファイル構成

```
data/
├── rfcs.json        # RFC一覧データ（メイン）
└── categories.json  # カテゴリ定義
```

### 3.3 rfcs.json スキーマ

```json
[
  {
    "number": 791,
    "name": "Internet Protocol",
    "shortName": "IP",
    "category": "network-layer",
    "obsoletedBy": null,
    "obsoletes": null,
    "note": ""
  },
  {
    "number": 9110,
    "name": "HTTP Semantics",
    "shortName": "HTTP",
    "category": "application-layer",
    "obsoletedBy": null,
    "obsoletes": [2616, 7230, 7231, 7232, 7233, 7235],
    "note": "HTTP/1.1の現行仕様"
  },
  {
    "number": 793,
    "name": "Transmission Control Protocol",
    "shortName": "TCP",
    "category": "transport-layer",
    "obsoletedBy": [9293],
    "obsoletes": null,
    "note": "TCPの原典。RFC 9293に廃止された。"
  }
]
```

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `number` | integer | ✓ | RFC番号 |
| `name` | string | ✓ | RFC正式名称 |
| `shortName` | string | | 略称・通称（任意。省略時は空文字 `""` を使用） |
| `category` | string | ✓ | カテゴリID（categories.jsonを参照）|
| `obsoletedBy` | integer[] or null | | このRFCを廃止した新しいRFC番号リスト（複数廃止RFC対応） |
| `obsoletes` | integer[] or null | | このRFCが廃止した旧RFC番号リスト |
| `note` | string | | 補足説明（任意。省略時は空文字 `""` を使用） |

### 3.4 categories.json スキーマ

```json
[
  { "id": "application-layer",  "label": "アプリケーション層" },
  { "id": "transport-layer",    "label": "トランスポート層" },
  { "id": "network-layer",      "label": "ネットワーク層" },
  { "id": "security",           "label": "セキュリティ" },
  { "id": "infrastructure",     "label": "インフラ・管理" },
  { "id": "other",              "label": "その他" }
]
```

**カテゴリID命名規則:**
- OSI/TCP-IPモデルの層を表すカテゴリは `{層名}-layer` 形式（ハイフン区切り複合語）
- 層以外のカテゴリは単語のみ

### 3.5 初期収録RFC（案）

代表的なRFCを優先的に収録する。廃止済みRFCは （廃止済） と明記する。

**アプリケーション層**
- RFC 9110: HTTP Semantics（現行）
- RFC 9112: HTTP/1.1（現行）
- RFC 9113: HTTP/2（現行）
- RFC 9114: HTTP/3（現行）
- RFC 7540: HTTP/2 （廃止済、RFC 9113に廃止）
- RFC 7230: HTTP/1.1 Message Syntax and Routing （廃止済、RFC 9110/9112に廃止）
- RFC 7231: HTTP/1.1 Semantics and Content （廃止済、RFC 9110に廃止）
- RFC 7232: HTTP/1.1 Conditional Requests （廃止済、RFC 9110に廃止）
- RFC 7233: HTTP/1.1 Range Requests （廃止済、RFC 9110に廃止）
- RFC 7234: HTTP/1.1 Caching （廃止済、RFC 9111に廃止）
- RFC 7235: HTTP/1.1 Authentication （廃止済、RFC 9110に廃止）
- RFC 1034: Domain Names - Concepts and Facilities
- RFC 1035: Domain Names - Implementation and Specification
- RFC 5321: SMTP

**トランスポート層**
- RFC 793: TCP （廃止済、RFC 9293に廃止）
- RFC 9293: Transmission Control Protocol（現行）
- RFC 768: UDP
- RFC 9000: QUIC

**ネットワーク層**
- RFC 791: IP (IPv4)
- RFC 8200: IPv6
- RFC 792: ICMP

**セキュリティ**
- RFC 8446: TLS 1.3
- RFC 6749: OAuth 2.0
- RFC 7519: JWT

**インフラ・管理**
- RFC 2119: Key words for use in RFCs（MUST, SHOULDなど）
- RFC 8174: Ambiguity of Uppercase vs Lowercase RFC 2119 Key Words

> **廃止済みRFCの掲載意図:** 過去仕様との対比学習、および廃止された番号自体を覚えることが実務・試験対策上有用なため収録する。デフォルトのクイズ出題対象からは除外される（セクション2.1「廃止済みRFCの出題方針」参照）。

## 4. 非機能要件

### 4.1 動作環境
- **ブラウザ**: モダンブラウザ（Chrome, Firefox, Safari, Edge の最新版）
- **デバイス**: PC・スマートフォン・タブレット対応（レスポンシブ）
- **オフライン**: 非対応（初回ロード時にJSONを取得）

### 4.2 パフォーマンス
- 初回表示: 3秒以内（GitHub Pages CDN経由）
- データ量: RFC数100件程度を想定（将来的に300件まで拡張可）

### 4.3 アクセシビリティ
- キーボード操作対応（クイズの回答選択・次の問題へ進む操作）
- 適切なコントラスト比

### 4.4 データ永続化

#### localStorageのデータ構造（キー: `rfc-tangocho-progress`）

```json
{
  "rfcs": {
    "793": { "correct": 3, "incorrect": 1, "lastSeen": "2026-02-27T10:00:00Z" },
    "9110": { "correct": 0, "incorrect": 2, "lastSeen": "2026-02-26T08:30:00Z" }
  },
  "lastQuizAt": "2026-02-27T10:05:00Z"
}
```

- キーはRFC番号を文字列化したもの
- `correct`: クイズでの正解回数（フラッシュカードの「覚えた」もカウント）
- `incorrect`: クイズでの不正解回数
- `lastSeen`: 最後にクイズ問題として出題された日時（ISO 8601形式）
- `lastQuizAt`: 最後にクイズを実施した日時（ISO 8601形式）
- ブラウザをまたいでの同期は対象外

### 4.5 学習進捗の利活用シナリオ

| 利用場所 | 表示・活用内容 |
|---|---|
| トップ画面 | 全体の進捗サマリー（例：「32件中18件習得済み」） |
| クイズ設定画面 | 「苦手問題のみ出題」オプション（不正解率50%超のRFCを優先出題） |
| クイズ結果画面 | 今回の正解数/出題数、累計正解率の変化 |
| 一覧・検索画面 | 各RFCの正解回数/挑戦回数を表示 |
| フラッシュカード | 「覚えた」を押した際にlocalStorageの `correct` をインクリメント |

## 5. 技術スタック

### 5.1 推奨構成

| 項目 | 採用技術 | 理由 |
|---|---|---|
| HTML/CSS | HTML5 + CSS3 | 標準技術、依存なし |
| JavaScript | バニラJS（ES Modules） | ビルドステップ不要、GitHub Pagesと相性良好 |
| データ形式 | JSON | 可読性高く、PR管理に適する |
| ホスティング | GitHub Pages | 静的サイト配信、無料 |

### 5.2 ディレクトリ構成（案）

```
rfc-tangocho/
├── index.html          # エントリーポイント
├── css/
│   └── style.css       # スタイルシート
├── js/
│   ├── app.js          # アプリケーションエントリ・ルーティング
│   ├── quiz.js         # クイズロジック（出題・採点・進捗記録）
│   ├── flashcard.js    # フラッシュカードロジック
│   ├── search.js       # 一覧・検索・フィルタロジック
│   ├── data.js         # JSONロード・管理・エラーハンドリング
│   └── storage.js      # localStorage操作（進捗CRUD）
├── data/
│   ├── rfcs.json       # RFC一覧データ
│   └── categories.json # カテゴリ定義
├── docs/
│   └── requirements.md # 本要件定義書
└── README.md
```

### 5.3 GitHub Pages デプロイ方法
- GitHub リポジトリの Settings > Pages で `main` ブランチのルートディレクトリ（`/`）を公開元に設定
- `main` ブランチへのマージが即時反映される
- GitHub Actions は使用しない（ビルド不要のため設定不要）

## 9. エラーハンドリング仕様

### 9.1 JSON fetchエラー

| エラー種別 | 発生条件 | ユーザーへの表示 | 挙動 |
|---|---|---|---|
| ネットワークエラー | fetch自体が失敗（オフライン等） | 「データの読み込みに失敗しました。ネットワーク接続を確認してください。」 | リトライボタンを表示 |
| HTTPエラー | fetch成功だが4xx/5xxレスポンス | 「データファイルが見つかりません（エラーコード: {status}）。」 | リトライボタンを表示 |
| JSONパースエラー | レスポンスが不正なJSON | 「データファイルの形式が正しくありません。」 | リトライ不可 |

### 9.2 データ整合性エラー

- `rfcs.json` 内の `category` フィールドの値が `categories.json` に存在しないIDを参照している場合: 該当RFCの `category` を `"other"` にフォールバックしてロード継続（コンソールに警告出力）
- `rfcs.json` 内のRFC件数が0件の場合: 「データが登録されていません。」メッセージを全モードで表示

### 9.3 localStorageエラー

- `localStorage` が利用不可（プライベートブラウジング等）の場合: 学習進捗機能を無効化し、クイズ・フラッシュカード・一覧は引き続き利用可能とする。ユーザーには「学習進捗の保存は無効です（プライベートブラウジング等）」と一度だけ通知する。
